# PrintPrice OS — Canonical Architecture

**Status:** Canonical Reference  
**Purpose:** Define the authoritative architecture of PrintPrice OS, including repository boundaries, runtime topology, control plane responsibilities, worker specialization, and federated printer connectivity.

---

## 1. Canonical Platform Overview

PrintPrice OS is a **distributed production network operating system**.

It is composed of:

- a **Product App** for end users
- a **Control Plane** for operators
- a **Core Platform** for orchestration and economics
- specialized **workers and services**
- a **federated printer network**
- shared **contracts** and **infra**
- a local **bootstrap/integration workspace**

The Product App is **not** the operating system.  
It is a consumer of the OS through APIs, contracts, and thin bridges.

## 3. Product App Boundary Contract

The Product App exists to serve **end users**.

It may contain:

* product UI / UX
* upload flows
* user authentication/session
* job submission and viewing
* thin BFF / ingress routes
* lightweight adapters to OS APIs
* product-side static assets

It must **not** contain:

* governance runtime
* SLO evaluation loops
* federation registry runtime
* printer matching logic
* redispatch logic
* printer health logic
* worker implementations
* control plane runtime
* printer agent code
* background governance loops
* operator-only platform logic

### Boundary Rule

If a piece of code:

* runs in background
* governs platform behavior
* talks to printers
* orchestrates dispatch
* executes heavy jobs
* or is meant for operators

then it **does not belong** in the Product App.

## 7. Repository Responsibilities

### `printprice-os-bootstrap`

Role:

* bootstrap
* local integration
* seeding
* initialization
* multi-repo workspace utilities

### `ppos-shared-infra`

Role:

* infra libraries
* Redis/MySQL utilities
* shared resilience/governance helpers where truly shared
* Docker/bootstrap support as needed

### `ppos-preflight-engine`

Role:

* deterministic PDF inspection
* technical extraction
* PDF safety logic
* print-spec validation

### `ppos-preflight-worker`

Role:

* background execution
* subprocess orchestration
* runtime leases / heartbeats
* heavy compute execution

### `ppos-preflight-service`

Role:

* HTTP service layer for preflight capabilities
* service-side ingress to engine/worker functionality

### `ppos-core-platform`

Role:

* platform brain
* job registry
* economics policy
* matchmaking
* federation dispatch orchestration

### `ppos-shared-contracts`

Role:

* DTOs
* JSON schemas
* TypeScript interfaces
* event contracts
* canonical payloads

### `ppos-build-orchestrator`

Role:

* build pipelines
* release promotion
* CI/CD workflows

### `ppos-governance-assurance`

Role:

* governance
* policy validation
* SLO evaluation
* resilience loops
* rollback guardrails
* runtime assurance

Role:

* remote printer connector
* heartbeat
* polling

### `preflight-app`

* end-user product

## 12. Integration Workspace Rule

A local multi-repo workspace is allowed and useful.

However:

* it is a **development integration workspace**
* it is **not** the canonical Product App repo
* it must not redefine ownership boundaries
* it must not justify placing platform runtime inside the product surface

---

## 13. Repository Boundary Rules

### Rule 1

The Product App consumes the OS.
It does not implement the OS.

### Rule 2

Background loops, schedulers, redispatchers, federation runtime, governance logic, and worker execution belong to OS repos, not Product App.

### Rule 3

Printer-facing logic belongs to federation/agent/runtime repos, not Product App.

### Rule 4

Operator-facing logic belongs to the Control Plane and governance repos, not Product App.

### Rule 5

Any code that can be classified as:

* control plane
* worker runtime
* federation runtime
* governance runtime
* agent runtime

must have a canonical home outside the Product App.

---

## 14. Deployment Boundary Rules

### Product App

Deploys independently to:

* `preflight.printprice.pro`

### Control Plane

Deploys independently to:

* `control.printprice.pro`

### Worker / Service / Agent repos

Deploy independently according to role:

* workers
* services
* agents
* governance loops

### Bootstrap

Is not the product deployment unit.

---

## 15. Canonical Success Condition

The architecture is considered healthy when:

* the Product App repo is clean and detached
* each OS repo owns its runtime responsibilities
* the control plane is independent
* workers are independent
* federation is independent
* the integration workspace remains useful but non-authoritative
* no new features violate these boundaries

---

## 16. Final Canonical Rule

```text
If a module governs, orchestrates, dispatches, executes, reconciles,
or serves operators or printer nodes, it does not belong in the Product App.
```
